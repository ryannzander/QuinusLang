Unsafe & Assembly

Hazard Blocks

Operations that bypass safety guarantees must be inside a hazard block:

hazard {
    // Unsafe code here
    reach raw_ptr = value;
}

This makes it explicit that the code may have undefined behavior if used incorrectly.

Inline Assembly: machine

Emit raw assembly instructions with machine blocks inside hazard:

hazard {
    machine { "mov eax, 1" };
    machine { "nop" };
}

Each string is one assembly instruction. The syntax is GCC-style (__asm__ __volatile__).

Platform Support

Inline assembly is supported when using GCC or Clang. On MSVC x64, inline assembly is not supported; the compiler emits a comment and the instruction is skipped. Use MinGW or Clang on Windows for assembly support.

Use Cases

  • Direct hardware access (memory-mapped I/O)
  • CPU-specific instructions (SIMD, crypto)
  • Kernel and bootloader code
  • Performance-critical inner loops

Safety

There are no runtime checks in hazard blocks. Incorrect assembly can crash the program or cause undefined behavior. Document constraints and invariants when using hazard.
